Propuesta técnica

duki sistema de socios + consumo + wallet + precios dinámicos

Resumen

Propuesta para implementar un portal interno que permita identificar socios mediante NFC, aplicar menús y precios por categoría, realizar cobros/recargas y mantener historial. Se propone iniciar con un MVP mínimo (flujo simple y rápido) y escalar por fases.

Alcance funcional (MVP)

  • Socios: alta/edición, categoría, estado (activo/bloqueado), credenciales (PIN), liga con NFC (UID cuando aplique).
  • Catálogo: productos (bebidas/alimentos) y disponibilidad por categoría.
  • Precios dinámicos: descuento por categoría + reglas para promociones futuras.
  • Wallet: saldo recargable y gastable (recargas y consumos).
  • Cobros: efectivo / tarjeta / transferencia / wallet.
  • Multi-ubicación: 2 sucursales + centro de entrenamiento como ubicación de consumo.
  • Seguridad: login para staff, control de acceso por roles, auditoría (quién hizo qué y dónde).

Flujos principales

  • Identificar socio: escaneo NFC → login del socio (PIN) → vista del socio.
  • Recarga: identificar socio → monto → método → confirmación → saldo actualizado.
  • Consumo: identificar socio → menú filtrado → precio final aplicado → cobro → historial.

NFC: estrategia y seguridad

NFC se considera un identificador / disparador de flujo. La protección de datos se realiza mediante credenciales del socio (PIN) y permisos del staff.

  • V1: escaneo NFC abre el login; el PIN controla el acceso.
  • Fase de hardening: evitar “links compartidos” mediante token temporal (10 min) / one-time, requiere backend mínimo.
  • Nota: un enlace NDEF puede ser copiado; la mitigación real necesita validación en servidor o aplicación.

Arquitectura propuesta por fases

  • Fase 0 (demo): mock sin backend para validar flujo y UX.
  • Fase 1 (MVP): backend + base de datos (SQLite o Postgres) para operar wallet, recargas y ventas con auditoría.
  • Fase 2 (hardening): tokens temporales, rate-limit de intentos, auditoría avanzada, backups automatizados.
  • Fase 3: reportes, promociones avanzadas, mejoras de operación multi-ubicación y roles finos.

Recomendación de implementación (MVP real)

  • Frontend: web app (tablet/PC) para staff + vistas para socio si aplica.
  • Backend: API modular (monolito modular) para mantener simple y escalable.
  • Datos: wallet con ledger (transacciones) para evitar inconsistencias.
  • Despliegue: un entorno central (o por sucursal) según conectividad y operación.
